In Agile methodology, ambiguity is not necessarily something that must be eliminated. However, unclarity, inconsistent features, missing requirements, and duplication are all consequences of ambiguity that must be reduced. The reason for this is that these issues can lead to technical debts which could compromise software quality.

Given these issues, we proposed the ambiguity analysis framework that is extracted based on a systematic review of the previous studies addressing ambiguity issues in user stories. The framework serves as a tool to identify potential ambiguity issues in a set of user stories by detecting user stories content based on their conformity with the framework quality criteria.

We further incorporate the framework with NLP techniques to allow users to independently analyze their set of user stories, express their opinion about the analysis results, and contribute to provide recommendations on how user stories should be improved.

However, this analysis is limitedly applied to user story that follow Connextra template and have been written in English. In addition, the analysis covers text-to-text analysis, meaning that the potential ambiguities identified and the recommendation of improvement that are suggested is limited to achieve a better understanding of user stories content in order to avoid the aforementioned problems.

Despite of the criteria of the framework that are depicted, the analysis only focus on Lexical, Syntactic, and Semantic levels.

The definition, situation for violation, and what we can do in the attempt of improvement are described as follows:

Req. Quality Criterion Definition Violation Recommendation for improvement
Clarity Precise terminology A situation where the actor/action in the WHO/WHAT segments are precisely described without allowing any other interpretations A situation where the actor/action in the WHO/WHAT segments are specified using broader terminologies which potentially results in different interpretations of action in WHAT and WHY segments Replace unclear/imprecise terminology with the standard glossary provided. When the standard glossary cannot give you sufficient recommendation, please add a new terminology to the library.
Atomicity A situation where a user story only deliver one functionality or feature. A situation where a user story potentially deliver more than one functionality or feature which could be observed from the presence of conjunctions. Split user story to avoid multiple functionalities or features delivered by a single user story
Conciseness A situation when user stories provide sufficient information about the expected functionality/feature A situation when user stories express a lot of information but still cannot clearly express what the system should deliver. Rephrase user stories to deliver sufficient information using simple sentence
Consistency Consistent terminology A condition when user stories are expressed using consistent terminology to express the same actor in the WHO segment or similar behavior in the WHAT and WHY segments A condition when user stories are expressed using different terminology to express the same actor in the WHO segment or similar behavior in the WHAT and WHY segments. Change inconsistent terminology with standard terminology provided in the glossary.
Well-formedness A condition when user story use a consistent template and contains a complete segment of WHO, WHAT, and WHY. A condition when user story does not written using a consistent template or does not have a complete segment of WHO, WHAT, and WHY. Change the structure of user story using Connextra template “As a [role], I want [goal], so that [benefit]”
Adequacy Conceptually sound A condition when user stories are sufficiently specified to deliver system requirements. A condition when user stories are overspecified or underspecified in order to convey system requirements. Rephrase user stories to avoid uoverspecified or underspecified action. Use terminology from the standard glossary to rephrase user stories.
Uniqueness Uniqueness A condition when one user story specified one specific functionality or feature. A condition when one functionality expected from a system are described in more than one user story. Remove one of two duplicated user stories.
